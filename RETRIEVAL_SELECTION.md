# Подбор retrieval для RAG-агента

Документ объясняет, **почему** в `02_rag_agent` выбраны конкретные методы, веса гибрида (α) и `top_k` для `search_by_chunks` и `search_by_titles`.

**Сырые цифры:** `01_retrieval_eval/results/report.md`, `title_search_report.md`, `pick_top_k.md`  
**Краткая сводка экспериментов:** [EXPERIMENTS.md](EXPERIMENTS.md)  
**Код конфигурации:** `02_rag_agent/agent_service/rag_agent/agent.py`

---

## 1. Что измеряли

| Параметр | Значение |
|----------|----------|
| Датасет | [Wix/WixQA](https://huggingface.co/datasets/Wix/WixQA) |
| Корпус в индексе | **4 172** документа (`article` + `known_issue`, без `feature_request`) |
| QA в метриках | **148** пар single-article (один gold `doc_id` на вопрос) |
| Прогон | `01_retrieval_eval/run_validation.py` |

Оценивается только **качество поиска** (нашли ли нужный документ/чанк в топ-K), не качество финального ответа LLM.

**Не входило в прогон:** chain-of-RAG (`run_eval.py --skip-chain` — нужен LLM API в бенчмарке).

### Метрики

**Чанки** (`search_by_chunks`):

- **MRR@K** — насколько высоко в выдаче первый релевантный чанк (главный критерий для ранжирования).
- **Recall@K** — доля запросов, где gold-чанк попал хотя бы в топ-K.

**Заголовки** (`search_by_titles`):

- **Hit@K** — доля запросов, где нужная **статья** есть в топ-K (главный критерий).
- **MRR@K** — позиция правильной статьи в списке.

### Что перебирали

- Методы: BM25, dense vector, hybrid linear (α от 0 до 1 с шагом 0.1), hybrid RRF (w ∈ {0.3, 0.5, 0.7}).
- Embed-модели (Ollama): `embeddinggemma`, `qwen3-embedding:0.6b`, `bge-m3`.
- K для метрик: 1, 3, 5, 6, 8, 10, 15, 20 (`EVAL_KS` в `.env`).

## 2. Итоговый конфиг в агенте

| Инструмент | Метод | Модель | α | top_k |
|------------|-------|--------|---|-------|
| `search_by_titles` | vector (dense по title) | embeddinggemma | **1.0** | **10** |
| `search_by_chunks` | hybrid linear (BM25 + vector по тексту чанка) | embeddinggemma | **0.7** | **8** |

В `rag_service` тот же embed и та же схема fusion: `ts_rank_cd` + pgvector `<=>`, min-max нормализация, linear fusion.

## 3. Поиск по чанкам

### 3.1. Почему hybrid, а не BM25 или чистый vector

| Метод | MRR@10 | Recall@10 |
|-------|--------|-----------|
| BM25 | 0.34 | 0.55 |
| vector (embeddinggemma) | 0.54 | **0.87** |
| **hybrid linear, α=0.7** | **0.58** | 0.87 |

- **BM25** слаб на перефразах и редких формулировках вопроса.
- **Vector** хорошо ловит смысл, Recall@10 максимален, но gold-чанк чаще оказывается **ниже** в топ-10 → хуже MRR.
- **Hybrid** добавляет лексический сигнал к семантическому: MRR@10 растёт на ~4 п.п. относительно vector при почти той же recall.

**RRF** (лучший вариант: embeddinggemma, w=0.7): MRR@10 = **0.50** — существенно ниже linear **0.58**. Для нашей нормализованной linear fusion RRF не дал выигрыша.

### 3.2. Почему embeddinggemma

При лучшем α для каждой модели (ориентир MRR@10):

| Модель | Лучший hybrid (ориентир) |
|--------|--------------------------|
| qwen3-embedding:0.6b | ~0.51 @ α=0.7 |
| bge-m3 | ~0.54 @ α=0.7 |
| **embeddinggemma** | **0.58 @ α=0.7** |

Одна модель в проде = один индекс в PostgreSQL и совпадение с лидером бенчмарка.

### 3.3. Почему α = 0.7

Sweep hybrid linear + embeddinggemma (MRR@10):

```
α=0.0 (только BM25)  → 0.34
α=0.5                → 0.53
α=0.6                → 0.56
α=0.7                → 0.58  ← максимум на сетке
α=0.8                → 0.57
α=1.0 (только vector)→ 0.54
```

Кривая смещена к **dense**: оптимум около 70% vector / 30% BM25. После 0.7 MRR слегка падает; у α=0.8 Recall@10 чуть выше (0.87 vs 0.86), но MRR хуже — для агента приоритет **ранга**, а не «хоть где-то в десятке».

α=0.6 был близок (0.562); выбран **0.7** как явный максимум на переборе.

### 3.4. Почему top_k = 8 (а не 15)

Кривая при hybrid α=0.7 (embeddinggemma):

| k | Recall | MRR |
|---|--------|-----|
| 5 | 0.79 | 0.57 |
| **8** | **0.83** | **0.57** |
| 10 | 0.87 | 0.58 |
| 15 | 0.93 | 0.58 |
| 20 | 0.95 | 0.58 |

**Наблюдения:**

1. После **k=8** MRR почти не растёт (+0.004 до k=15), recall ещё прибавляет ~10 п.п.
2. Каждый чанк — текст в контекст LLM; 15 чанков заметно дороже 8 по токенам и шуму.
3. Формальный критерий «≥95% от max recall на сетке k» при α=0.7 даёт **k=15** (recall@15 = 0.926).

**Решение для агента:** default **top_k=8** — компромисс «почти лучший MRR + приемлемый recall»; при необходимости tool допускает до **15** (`max(1, min(top_k, 15))`).

| Цель | top_k |
|------|-------|
| Баланс MRR / размер контекста (default) | **8** |
| Максимизировать recall по бенчмарку | 15 |

## 4. Поиск по заголовкам

### 4.1. Почему vector (α = 1.0)

| Метод | Hit@10 | MRR@10 |
|-------|--------|--------|
| BM25 по title | 0.29 | 0.18 |
| **vector / hybrid α=1.0** | **0.74** | **0.46** |

Заголовки короткие; вопрос пользователя часто **не совпадает лексически** с title статьи. Dense search это закрывает лучше BM25.

Hybrid с α &lt; 1.0 **не улучшает** Hit@10: BM25 по title только ухудшает или не меняет метрики. Лучшие `hybrid_title` при α=0.9–1.0 совпадают с vector. В агенте **α=1.0** — чистый dense, проще в коде и на метриках не хуже hybrid.

### 4.2. Почему top_k = 10

| k | Hit | MRR |
|---|-----|-----|
| 5 | 0.62 | 0.45 |
| **10** | **0.74** | **0.46** |
| 15 | 0.80 | 0.47 |
| 20 | 0.82 | 0.47 |

- **Hit@10 = 0.743** — плато на кривой для данной конфигурации (максимум Hit@10).
- Рост Hit@15 всего +6 п.п., MRR +0.01 — слабый прирост.
- Tool жёстко ограничен **max 10** статей (`min(top_k, 10)`).

**k=10** совпадает с лимитом инструмента, метрикой Hit@10 и критерием «95% от max hit» на сетке k.

## 5. Два инструмента в одном агенте

```
Вопрос пользователя
       │
       ▼
search_by_titles (k=10, α=1.0)   ← уровень «какая статья»
       │
       ├──► open_article(id)
       │
       ▼
search_by_chunks (k=8, α=0.7)    ← уровень «какой фрагмент / шаг»
```

Разные поля индекса и разные метрики на бенчмарке:

- **Titles** — отбор документа целиком (Hit@10).
- **Chunks** — точные пассажи внутри статей (MRR@10).

Поэтому разумны **разные α и top_k**, а не один общий конфиг retrieval.

## 6. Что сознательно не выбрали

| Вариант | Почему нет |
|---------|------------|
| BM25 only | MRR@10 0.34 (чанки), Hit@10 0.29 (titles) |
| Vector only для чанков | MRR@10 0.54 &lt; hybrid 0.58 |
| RRF fusion | MRR@10 0.50 &lt; linear 0.58 |
| qwen / bge-m3 | хуже embeddinggemma на всех α |
| chain-of-RAG | не оценивался в финальном прогоне |
| top_k=15 для чанков по умолчанию | +recall ценой контекста; MRR после k=8 почти не растёт |
| top_k&gt;10 для titles | cap tool; слабый прирост Hit после 10 |

---

## 7. Ограничения выводов

1. **148 QA** с одним gold-документом — метрики не отражают multi-hop и вопросы без одного ответа в одной статье.
2. Бенчмарк на **Chroma + rank-bm25**; агент на **PostgreSQL** — та же логика fusion, другая реализация (допустимо для прототипа, не побиточное равенство).
3. Не измерялось end-to-end качество ответов агента после retrieval.
4. Корпус без `feature_request` — метрики на «инструктивных» статьях Wix Help Center.

## 8. Воспроизведение и обновление конфига

```bash
cd 01_retrieval_eval
python download_data.py
python run_validation.py
```

Пересчёт top_k после смены α:

```bash
python pick_top_k.py --chunks-alpha 0.7
```

После смены рекомендаций обновить:

- `02_rag_agent/agent_service/rag_agent/agent.py` — `_ALPHA_CHUNKS`, `_ALPHA_TITLES`, defaults `top_k`
