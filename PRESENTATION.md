## Слайд 1. Тема и цель

**Тема:** Сравнительный анализ методов поиска в системах Retrieval-Augmented Generation (RAG)

**Цель:** Определить оптимальную стратегию retrieval для корпоративной базы знаний и реализовать на её основе диалогового агента поддержки

**Задачи:**
- Провести бенчмарк методов поиска: BM25, векторный, гибридный (linear и RRF)
- Сравнить три модели эмбеддингов через Ollama
- Выбрать лучший конфиг по метрике MRR@10
- Реализовать прототип RAG-агента с веб-интерфейсом

## Слайд 2. Датасет — Wix WixQA

**Источник:** HuggingFace — `Wix/WixQA` (опубликован Wix Research, май 2025)

**Корпус:** 6 221 статья из Wix Help Center
- Поля: `id`, `title`, `contents`, `url`, `article_type`
- Медианная длина статьи: ~1 800 символов
- 64% статей содержат пошаговые инструкции

**QA-пары:** 200 вопросов, написанных экспертами поддержки
- Каждый вопрос связан с 1–3 эталонными статьями (`article_ids`)
- **Для оценки использовались только 148 пар с одной эталонной статьёй** — это обеспечивает однозначную метрику без завышения за счёт нескольких правильных ответов

## Слайд 3. Метрика оценки

**MRR@K (Mean Reciprocal Rank)** — основная метрика

$$MRR@K = \frac{1}{|Q|} \sum_{q=1}^{|Q|} \frac{1}{\text{rank}_q}$$

где $\text{rank}_q$ — позиция первого релевантного документа в топ-K для запроса $q$.

- MRR = 1.0: правильная статья всегда на первом месте
- MRR = 0.5: правильная статья в среднем на второй позиции
- MRR = 0.0: правильная статья не найдена

**Дополнительно:** Recall@K — доля запросов где правильная статья попала в топ-K

## Слайд 4. Методы поиска

### BM25 (полнотекстовый поиск)
Классический tf-idf с нормализацией длины документа. Строит инвертированный индекс по токенам. Быстрый, не требует GPU, хорошо работает на точных совпадениях ключевых слов.

### Векторный поиск (dense retrieval)
Документы и запросы кодируются в плотные векторы через языковую модель. Поиск ближайших соседей по косинусному расстоянию. Улавливает семантическое сходство.

### Гибридный поиск (hybrid)
Комбинирует BM25 и векторный поиск. Протестированы два метода слияния:

- **Linear fusion:** `score = α·vec + (1−α)·bm25`, нормализация min-max  
  Параметр α перебирался от 0.0 до 1.0 с шагом 0.1
- **RRF (Reciprocal Rank Fusion):** `score = w·rrf(vec_rank) + (1−w)·rrf(bm25_rank)`,  
  где `rrf(rank) = 1/(60 + rank)` — формула Cormack et al. 2009

### Модели эмбеддингов (Ollama, локально)
| Модель | Размерность | Query prefix |
|---|---|---|
| `embeddinggemma` | 768 | `task: search result \| query: {text}` |
| `qwen3-embedding:0.6b` | 1024 | `Instruct: ...\nQuery:{text}` |
| `bge-m3` | 1024 | без prefix |

## Слайд 5. Результаты — поиск по содержанию (чанки)

Оценка на 148 single-article QA парах.

### Лучшие конфигурации по MRR@10:

| Ранг | Метод | Модель | Alpha | MRR@1 | MRR@3 | MRR@10 | Recall@10 |
|---|---|---|---|---|---|---|---|
| 1 | Hybrid Linear | embeddinggemma | **α=0.6** | 0.446 | 0.534 | **0.570** | 0.858 |
| 2 | Hybrid Linear | embeddinggemma | α=0.7 | 0.378 | 0.502 | 0.541 | 0.858 |
| 3 | Hybrid RRF | embeddinggemma | w=0.7 | 0.365 | 0.467 | 0.520 | 0.865 |
| 4 | Vector | embeddinggemma | — | 0.345 | 0.453 | 0.501 | 0.865 |
| 5 | Hybrid Linear | bge-m3 | α=0.6 | 0.399 | 0.481 | 0.523 | 0.824 |
| — | BM25 | — | — | 0.264 | 0.332 | 0.361 | 0.588 |

**Вывод:** Гибридный линейный поиск с `embeddinggemma` и `α=0.6` даёт лучший результат. Чисто векторный поиск на 10% лучше BM25, гибрид ещё на 14% лучше векторного.

## Слайд 6. Результаты — поиск по заголовкам

Задача: по вопросу найти правильную статью, зная только её заголовок. Метрика — Hit@K (статья попала в топ-K).

| Ранг | Метод | Модель | Alpha | Hit@1 | Hit@5 | Hit@10 | MRR@10 |
|---|---|---|---|---|---|---|---|
| 1 | Vector | embeddinggemma | — | 0.338 | 0.595 | **0.730** | **0.453** |
| 2 | Hybrid | embeddinggemma | α=0.9 | 0.324 | 0.601 | 0.716 | 0.441 |
| 3 | Hybrid | embeddinggemma | α=0.6 | 0.277 | 0.547 | 0.710 | 0.401 |
| — | BM25 | — | — | 0.128 | 0.210 | 0.277 | 0.164 |

**Вывод:** Для поиска по заголовкам чисто векторный поиск незначительно лучше гибридного — BM25 по коротким заголовкам работает значительно хуже, чем по полным текстам. Hit@10 = 0.73 означает, что в 73% случаев правильная статья попадает в топ-10 результатов поиска по заголовку.

## Слайд 7. Выводы по бенчмарку

| Аспект | Результат |
|---|---|
| Лучший метод | Hybrid Linear, `embeddinggemma`, α=0.6 |
| MRR@10 (лучший) | **0.570** |
| Прирост к BM25 | +58% |
| Прирост к vector | +14% |
| Recall@10 (лучший) | 0.858 |

**Ключевые наблюдения:**
1. `embeddinggemma` значительно превосходит `bge-m3` и `qwen3-embedding:0.6b` на этом корпусе
2. Оптимальное α=0.6 — умеренный перевес в сторону векторного поиска, BM25 даёт важный сигнал точных совпадений
3. RRF незначительно уступает linear fusion (MRR@10 = 0.520 vs 0.570)
4. BM25 одного токена недостаточно: заголовки статей Wix Help Center семантически близки, ключевые слова часто совпадают в разных статьях

## Слайд 9. Ключевые технические решения

### Параграфный чанкинг
Вместо наивного разбиения по символам — разбивка по `\n`-параграфам с объединением коротких (< 100 символов). Целевой размер чанка ~600 символов. Длинные параграфы (стена текста) дополнительно режутся по предложениям. Перекрытие — последнее предложение предыдущего чанка.

### Гибридный поиск в SQL
Один запрос к PostgreSQL — BM25 через `ts_rank_cd(fts, plainto_tsquery(...))` и векторный через `<=> (cosine)` оператор pgvector. Min-max нормализация и linear fusion внутри CTE.

### Лучший чанк в результатах поиска по заголовкам
При поиске по заголовкам каждый результат содержит не первые 200 символов, а `DISTINCT ON (doc_id) ORDER BY chunk_vec <=> query_vec` — наиболее релевантный запросу фрагмент статьи.

### Персистентная память агента
Кастомный `PostgresMemoryService(BaseMemoryService)` — ADK после каждой сессии сохраняет диалог в таблицу `agent_memory`. При новом разговоре `preload_memory_tool` достаёт релевантные фрагменты через полнотекстовый поиск.

### Артефакты
При вызове `open_article` полный текст сохраняется в ADK artifact service. Агент может читать его построчно (`read_article_lines`) или искать по нему регулярными выражениями (`search_in_article`) — не запрашивая БД повторно.

## Слайд 10. Демонстрация

**Сценарий 1 — простой вопрос:**
> "Как подключить домен к Wix сайту?"

1. `search_by_titles("connect domain to Wix site")` → топ-10 статей с best_chunk
2. Агент видит релевантный чанк, отвечает по нему
3. Указывает источник: *"Connecting a Domain to Your Wix Site"*

**Сценарий 2 — сложный вопрос:**
> "Не работает форма оплаты, ошибка при checkout"

1. `search_by_titles(...)` → статьи о платёжных системах
2. `search_by_chunks("checkout error payment form")` → конкретные шаги отладки
3. `open_article(id)` → сохраняет статью в артефакт, показывает первые 80 строк
4. `search_in_article("error code", context_lines=2)` → находит секцию с кодами ошибок

**Сценарий 3 — повторный визит:**
> "Помнишь, мы обсуждали проблему с доменом?"

1. `preload_memory_tool` автоматически загрузил историю при старте
2. Агент ссылается на предыдущий разговор без повторного поиска

## Слайд 12. Заключение

**Результаты:**
- Проведён бенчмарк 46 конфигураций поиска на реальном enterprise-датасете Wix WixQA
- Лучший метод: гибридный линейный поиск, `embeddinggemma`, α=0.6, MRR@10 = **0.570**
- Реализован прототип диалогового агента с 7 инструментами, персистентной памятью и системой артефактов
- Агент корректно работает с длинными документами через построчное чтение и regex-поиск по артефактам

**Направления развития:**
- Файн-тюнинг модели эмбеддингов на доменных данных
- Reranker (cross-encoder) для финальной ранжировки
- Оценка качества генерируемых ответов (не только retrieval)
- A/B тестирование на реальных пользователях
